# 기능 1~3 학습 기록

## 오늘 학습한 내용
* 검색 시스템의 성능을 감으로 판단하지 않고, 질문과 정답 문서 ID로 구성된 평가셋을 이용해 수치로 측정하는 방법을 학습했다.
* Precision@k가 상위 k개의 검색 결과 중 정답 문서가 차지하는 비율이라는 점을 이해했다.
* Reciprocal Rank가 검색 결과에서 첫 번째 정답이 등장한 순위의 역수이며, 여러 질문의 Reciprocal Rank 평균이 MRR이라는 점을 학습했다.
* 검색 평가에서는 질문만 만드는 것이 아니라, 각 질문에 실제로 답이 되는 문서 ID를 직접 확인해 정답으로 지정하는 과정이 중요하다는 것을 확인했다.

## 구현한 기능
* Python, Git, AI 기초, NumPy, pandas 카테고리를 고르게 포함한 12개의 평가 질문을 `EVAL_SET`으로 구성했다.
* 각 평가 항목을 `query`와 `relevant_doc_ids`로 구성하고, 질문마다 1~2개의 실제 문서 ID를 정답으로 지정했다.
* 검색 결과 상위 k개와 정답 문서 목록의 교집합을 이용해 `Precision@k`를 계산하는 `precision_at_k()` 함수를 구현했다.
* 검색 결과를 1위부터 순회하며 처음 등장한 정답 순위의 역수를 반환하는 `reciprocal_rank()` 함수를 구현했다.
* 평가셋에 작성한 정답 문서 ID가 실제 데이터에 존재하는지 확인하는 검증 코드를 추가했다.
* Precision@3이 `0.6667`, 정답이 1위·2위·검색 결과에 없는 경우의 Reciprocal Rank가 각각 `1.0`, `0.5`, `0.0`인지 테스트했다.

## 막힌 부분 및 해결
* 실제 오류로 진행이 중단된 부분은 없었다.
* 다만 평가 질문의 정답 문서를 제목만 보고 무조건 하나로 지정하면 관련 문서를 놓칠 수 있어, 원문 내용을 함께 확인하고 질문에 따라 1~2개의 정답 문서를 지정했다.
* MRR과 Reciprocal Rank의 차이가 처음에는 혼동될 수 있어, 한 질문에 대해 계산하는 값은 Reciprocal Rank이고 여러 질문의 평균을 계산한 값이 MRR이라는 기준으로 구분했다.
* `reciprocal_rank()` 자체에는 `k` 인자가 없으므로, 이후 평가 단계에서는 검색 함수가 반환한 Top-k 문서 ID 목록을 전달하도록 설계해야 한다는 점을 정리했다.

## AI·LLM 검색과의 연결
* RAG 시스템에서는 생성 모델이 답변을 잘 작성하더라도 검색 단계에서 관련 문서를 가져오지 못하면 정확한 답변을 만들기 어렵다.
* Precision@k는 검색 결과 묶음 안에 관련 문서가 얼마나 포함되었는지를 보여주고, MRR은 가장 먼저 등장한 관련 문서의 순위가 얼마나 높은지를 보여준다.
* 따라서 이번에 구현한 지표는 이후 RAG 검색기, 문서 임베딩 모델, 재정렬 모델의 품질을 비교하는 기본 평가 도구로 활용할 수 있다.

# 기능 4~6 학습 기록

## 오늘 학습한 내용
* 평가셋 전체에 검색을 반복 실행하고, 각 질문에서 계산한 Precision@3와 Reciprocal Rank의 평균을 구해 검색기 전체 성능을 평가하는 방법을 학습했다.
* 질문 하나만 입력받는 공통 검색 함수 형태로 맞추면 서로 인자가 다른 Keyword Baseline과 TF-IDF 검색기도 동일한 `run_evaluation()` 함수로 평가할 수 있다는 것을 이해했다.
* Precision@3와 MRR은 서로 다른 관점의 지표이므로 한 검색기가 두 지표에서 모두 항상 높게 나오지는 않는다는 점을 확인했다.
* 실패 케이스 분석은 단순히 평균 점수만 확인하는 것이 아니라, 정답을 찾지 못한 질문과 실제 검색 결과를 직접 비교해 검색기의 한계를 찾는 과정이라는 것을 학습했다.

## 구현한 기능
* 평가셋의 각 질문을 검색 함수에 전달하고 검색 결과의 `doc_id` 목록을 가져오는 `run_evaluation()` 함수를 구현했다.
* 각 질문의 Precision@3와 Reciprocal Rank를 계산해 리스트에 저장하고, NumPy의 `mean()`으로 평균 Precision@3와 MRR을 계산했다.
* 필요한 인자가 서로 다른 `keyword_search()`와 `tfidf_search()`를 `lambda`로 감싸 질문 하나만 받는 동일한 형태로 통일했다.
* Keyword Baseline과 TF-IDF를 같은 평가셋으로 평가하고, 두 검색 방식의 Precision@3와 MRR을 pandas DataFrame 표로 비교 출력했다.
* Reciprocal Rank가 `0.0`인 질문을 찾아 질문, 정답 문서 ID, 실제 Top-3 검색 결과를 출력하는 `analyze_failures()` 함수를 구현했다.
* 데이터 로드, 결측치 처리, 텍스트 전처리, TF-IDF 벡터화, 두 검색기 평가, 성능 비교, 실패 분석을 순서대로 실행하는 `main()` 함수를 구현했다.
* `if __name__ == "__main__":` 블록을 추가해 `python week03/main.py` 명령으로 전체 프로그램이 실행되도록 연결했다.
* 평가셋에 작성한 모든 정답 문서 ID가 실제 데이터에 존재하는지 확인하는 검증을 최종 실행 과정에 포함했다.

## 실행 결과
* 평가셋 크기: 12개 질문
* TF-IDF 행렬 크기: 60개 문서 × 401개 단어 특성
* Keyword Baseline Precision@3: `0.3889`
* Keyword Baseline MRR: `1.0000`
* TF-IDF Precision@3: `0.4167`
* TF-IDF MRR: `0.9028`
* TF-IDF Top-3 실패 케이스: `0개`

## 결과 해석
* TF-IDF의 Precision@3가 Keyword Baseline보다 높아, 상위 3개 결과 전체를 기준으로 보면 관련 문서를 조금 더 많이 포함했다.
* Keyword Baseline은 모든 질문에서 정답 문서 하나를 1위에 배치해 MRR이 `1.0000`으로 나왔다.
* TF-IDF는 Git 브랜치 병합 질문의 정답 `D014`를 2위에 배치하고, 신경망 계층의 gradient 질문의 정답 `D030`을 3위에 배치해 MRR이 `0.9028`로 낮아졌다.
* TF-IDF가 무조건 모든 지표에서 더 높아야 하는 것은 아니며, 평가 질문과 정답 구성에 따라 Precision과 MRR의 결과가 다르게 나타날 수 있음을 확인했다.

## 막힌 부분 및 해결
* Keyword Baseline과 TF-IDF 검색 함수가 요구하는 인자 수가 달라 `run_evaluation()`에서 같은 방식으로 호출하기 어려웠다.
* 두 검색 함수를 `lambda query: ...` 형태로 감싸 질문 하나만 받도록 통일하여 하나의 평가 함수를 재사용할 수 있도록 해결했다.
* 처음에는 과제 예시처럼 TF-IDF의 MRR이 더 높지 않아 구현이 잘못된 것인지 의심했지만, 질문별 검색 순위를 출력해 확인한 결과 TF-IDF가 일부 정답을 2위와 3위에 배치했기 때문임을 확인했다.
* TF-IDF의 Top-3 실패 케이스가 한 건도 나오지 않았지만, 평가셋에 따라 실패가 없을 수도 있으므로 오류는 아니다. 실패가 없을 때도 프로그램이 `실패 케이스가 없습니다.`라고 출력하도록 처리했다.

## AI·LLM 검색과의 연결
* RAG 시스템에서는 검색기가 관련 문서를 얼마나 많이 가져오는지뿐 아니라, 가장 중요한 정답 문서를 얼마나 앞 순위에 배치하는지도 답변 품질에 직접 영향을 준다.
* Precision@k는 모델에 전달되는 검색 문서 묶음의 관련성 밀도를 확인하는 지표이고, MRR은 사용자가 가장 먼저 보거나 LLM이 우선적으로 활용할 정답 문서의 순위를 확인하는 지표다.
* 두 지표를 함께 사용하면 검색기가 여러 관련 문서를 넓게 찾는지와 핵심 정답을 상단에 배치하는지를 구분해 평가할 수 있다.
* 이후에는 키워드가 정확히 겹치지 않는 질문을 처리하기 위해 임베딩 기반 의미 검색, Keyword와 벡터 검색을 결합한 Hybrid Search, 검색 결과를 다시 정렬하는 Re-ranking 등을 적용하고 현재 점수와 비교할 수 있다.
