# 기능 1 학습 기록

## 오늘 학습한 내용
* 1~3주차에서 각각 구현한 데이터 로드, 텍스트 전처리, TF-IDF 벡터화, 검색, 평가, 오류 분석 기능을 하나의 실행 흐름으로 연결하는 방법을 학습했다.
* 개별 함수가 정상적으로 구현되어 있어도 실제 프로그램에서는 입력 데이터와 중간 결과가 다음 단계에 올바르게 전달되도록 호출 순서를 설계해야 한다는 점을 이해했다.
* if __name__ == "__main__": 블록을 사용하면 파일이 직접 실행될 때만 main()이 호출되며, 다른 파일에서 함수만 가져올 때는 전체 파이프라인이 자동 실행되지 않는다는 점을 확인했다.
* 동일한 TF-IDF 행렬과 vectorizer를 검색과 평가에서 재사용해야 문서와 질문이 같은 벡터 공간에서 비교된다는 점을 복습했다.

## 구현한 기능
* 3주차의 main.py를 week04/main.py로 복사하고, 과제 1~3에서 작성한 모든 함수를 한 파일에 유지했다.
* main()에서 문서 로드 → 결측치 처리 → 전처리 → 평가셋 검증 → TF-IDF 벡터화 순서로 실행되도록 연결했다.
* git merge conflicts 예시 질문으로 tfidf_search()를 실행하고, 상위 3개 문서의 doc_id, 제목, 카테고리, 코사인 유사도가 출력되도록 구현했다.
* Keyword Baseline과 TF-IDF를 동일한 12개 평가 질문으로 실행하고 Precision@3와 MRR을 표로 비교하도록 연결했다.
* TF-IDF 검색 결과에서 정답 문서가 Top-3 안에 없는 질문을 analyze_failures()로 출력하도록 연결했다.
* if __name__ == "__main__": 블록에서 main()을 호출해 python week04/main.py 명령 한 번으로 전체 파이프라인이 실행되도록 구성했다.

## 막힌 부분 및 해결
* 3주차 코드에도 이미 main()이 있었기 때문에 과제 4에서 무엇을 새로 해야 하는지 처음에는 구분하기 어려웠다.
* 과제 4 기능 1의 핵심은 새로운 평가 기능을 추가하는 것이 아니라, 예시 검색 단계를 포함해 로드 → 벡터화 → 검색 → 평가 → 오류 분석의 실행 순서를 명확히 보여주는 것이라고 정리했다.
* 공식 예시는 week4 폴더를 사용하지만 저장소의 기존 규칙과 통일하기 위해 week04 폴더를 사용했다.
* 데이터 경로가 프로젝트 루트 기준인 data/tech_docs.csv이므로 week04 폴더 안에서 실행하지 않고 프로젝트 루트에서 python week04/main.py로 실행하도록 정리했다.

## 실행 확인
* data/tech_docs.csv가 정상적으로 로드되는지 확인한다.
* TF-IDF 행렬 크기와 사용된 단어 수가 출력되는지 확인한다.
* git merge conflicts 예시 검색 결과 3개가 출력되는지 확인한다.
* Keyword Baseline과 TF-IDF의 Precision@3·MRR 비교표가 출력되는지 확인한다.
* TF-IDF 실패 케이스 또는 실패 케이스가 없습니다. 메시지가 출력되는지 확인한다.

## AI·LLM 검색과의 연결
* 이번 통합 파이프라인은 RAG에서 문서를 불러오고 검색 가능한 형태로 변환한 뒤, 사용자 질문과 관련된 문서를 찾고 검색 품질을 평가하는 기본 구조와 같다.
* 현재의 TF-IDF 벡터를 임베딩 벡터로 바꾸고, 직접 계산한 유사도 검색을 Vector DB 검색으로 바꾸면 실제 RAG 검색 파이프라인으로 확장할 수 있다.
* 검색 결과를 생성 모델에 전달하기 전에 Precision@k, MRR, 실패 케이스를 확인하는 과정은 RAG 답변 품질 저하의 원인이 검색 단계인지 생성 단계인지 구분하는 데 도움이 된다.
